# Multica 自托管接入公司 CAS SSO 改造方案

## 1. 背景

当前需要基于 `multica-ai/multica` 做自托管部署，并接入公司的统一登录体系 CAS SSO。

Multica 当前官方 self-host 登录方式主要包括：

1. 邮箱 + 6 位验证码登录；
2. Google OAuth 登录。

登录成功后，服务端会签发 JWT cookie，Web UI 依赖该 cookie 完成登录态维护。Multica 还存在多类 token，包括 Web 使用的 JWT cookie、CLI/PAT 使用的 Personal Access Token、daemon 使用的 daemon token。

因此，本次改造不应绕过 Multica 原有认证体系，也不应只在外部 SSO 网关写入 DB 或缓存。正确方案是：**新增 CAS 登录入口，把 CAS 验证后的公司用户信息归一化为 Multica 用户，然后复用现有用户、JWT、cookie、CLI token、daemon token 链路。**

---

## 2. 总体目标

完成公司 CAS SSO 接入，使公司员工可以通过公司 SSO 登录自托管 Multica。

目标包括：

1. 新增公司 CAS 登录入口；
2. CAS 登录成功后，自动创建或查找 Multica 用户；
3. 复用现有 Multica JWT/cookie 机制；
4. 保持 CLI login、daemon、PAT 相关链路可用；
5. 支持禁用邮箱验证码登录和 Google 登录；
6. 支持限制仅公司用户登录；
7. 为后续企业账号生命周期管理预留能力，例如账号禁用、离职停用。

---

## 3. 非目标

第一阶段不要做过度重构。

暂不做：

1. 不重写完整认证体系；
2. 不引入独立 Redis session 体系；
3. 不直接由外部 SSO 网关写 Multica 登录态；
4. 不绕过 Multica `issueJWT`、`SetAuthCookies`、PAT、daemon token 体系；
5. 不重构成完整 OAuth/OIDC Provider Registry；
6. 不支持多个外部 IdP；
7. 不实现复杂 RBAC / 组织架构 / 部门同步。

第一阶段只做公司 CAS 登录闭环。

## 5. 现有认证链路分析

### 5.1 用户查找与创建

当前后端已有公共函数 `findOrCreateUser(ctx, email)`，负责根据 email 查找或创建用户。新用户会走 signup allowlist 校验，已有用户直接返回。

CAS 登录成功后，应复用该函数。

### 5.2 JWT 签发

当前后端已有 `issueJWT(user)`，会签发包含 `sub/email/name/exp/iat` 的 JWT。

CAS 登录成功后，应复用该函数，不要自己另写 token 体系。

### 5.3 Cookie 写入

邮箱验证码登录和 Google OAuth 登录成功后，都会调用 `auth.SetAuthCookies(w, tokenString)` 设置 Web 登录 cookie。

CAS callback 也应复用该逻辑。

### 5.4 CLI 登录

Multica 登录页已有 CLI callback 逻辑。浏览器登录成功后，可以把 token 回传给本地 CLI callback。

后端也已有 `IssueCliToken`，会基于当前已认证用户重新签发 token 给 CLI 使用。

因此，CAS 登录必须兼容 CLI login，不要只处理普通 Web 页面登录。

### 5.5 当前登录页

当前登录页主要支持邮箱验证码和 Google 登录。Google 按钮由 `google` 配置或 `onGoogleLogin` 控制展示。

需要新增公司 SSO 登录按钮，并支持运行时配置。

## 6. 目标登录流程

### 6.1 Web 登录流程

```txt
用户访问 Multica 登录页
  ↓
点击“使用公司 SSO 登录”
  ↓
跳转 /api/auth/cas/start
  ↓
后端重定向到公司 CAS 登录页
  ↓
用户在 CAS 完成登录
  ↓
CAS 携带 ticket 回跳 /api/auth/cas/callback
  ↓
Multica 后端校验 ticket
  ↓
解析公司用户信息
  ↓
归一化为 email/name/avatar
  ↓
findOrCreateUser(ctx, email)
  ↓
issueJWT(user)
  ↓
auth.SetAuthCookies(w, token)
  ↓
重定向回 Multica 前端
  ↓
Web 登录成功
```

### 6.2 CLI 登录流程

```txt
用户执行 multica login
  ↓
CLI 打开浏览器登录页，携带 cli_callback
  ↓
用户点击公司 SSO 登录
  ↓
CAS 登录成功
  ↓
Multica 设置 Web cookie
  ↓
登录页识别 cli_callback
  ↓
调用 IssueCliToken 或直接使用 token
  ↓
重定向回本地 CLI callback
  ↓
CLI 保存 token
  ↓
CLI / daemon 正常工作
```

## 7. 环境变量设计

新增以下环境变量。

```env
CAS_ENABLED=true
CAS_DISPLAY_NAME=Company SSO

CAS_LOGIN_URL=https://sso.company.com/cas/login
CAS_VALIDATE_URL=https://sso.company.com/cas/serviceValidate
CAS_SERVICE_URL=https://multica.company.com/auth/cas/callback

CAS_ATTRIBUTE_EMAIL=email
CAS_ATTRIBUTE_NAME=name
CAS_ATTRIBUTE_AVATAR=avatar

CAS_EMAIL_DOMAIN=company.com
```

说明：

| 变量 | 说明 |
|---|---|
| `CAS_ENABLED` | 是否启用 CAS 登录 |
| `CAS_DISPLAY_NAME` | 前端展示名称，例如“公司 SSO” |
| `CAS_LOGIN_URL` | CAS 登录地址 |
| `CAS_VALIDATE_URL` | CAS ticket 校验地址 |
| `CAS_SERVICE_URL` | CAS 回调 service 地址，必须与 CAS 服务端配置一致 |
| `CAS_ATTRIBUTE_EMAIL` | 从 CAS attributes 中读取 email 的字段名 |
| `CAS_ATTRIBUTE_NAME` | 从 CAS attributes 中读取 name 的字段名 |
| `CAS_ATTRIBUTE_AVATAR` | 从 CAS attributes 中读取头像字段名，可选 |
| `CAS_EMAIL_DOMAIN` | 当 CAS 不直接返回 email 时，用 username 拼接邮箱 |

可选新增：

```env
EMAIL_LOGIN_ENABLED=false
GOOGLE_LOGIN_ENABLED=false
```

说明：

| 变量 | 说明 |
|---|---|
| `EMAIL_LOGIN_ENABLED` | 是否启用邮箱验证码登录 |
| `GOOGLE_LOGIN_ENABLED` | 是否启用 Google 登录 |

公司内部部署建议：

```env
APP_ENV=production
CAS_ENABLED=true
EMAIL_LOGIN_ENABLED=false
GOOGLE_LOGIN_ENABLED=false
ALLOWED_EMAIL_DOMAINS=company.com
ALLOW_SIGNUP=false
JWT_SECRET=<强随机值>
FRONTEND_ORIGIN=https://multica.company.com
```

## 8. 后端改造

### 8.1 新增 CAS 配置结构

在后端 config 中新增 CAS 相关字段。

建议结构：

```go
type CASConfig struct {
    Enabled         bool
    DisplayName     string
    LoginURL        string
    ValidateURL     string
    ServiceURL      string
    AttributeEmail  string
    AttributeName   string
    AttributeAvatar string
    EmailDomain     string
}
```

启动时从环境变量读取。

需要校验：

```txt
CAS_ENABLED=true 时，CAS_LOGIN_URL、CAS_VALIDATE_URL、CAS_SERVICE_URL 必填
CAS_SERVICE_URL 必须是合法 URL
CAS_LOGIN_URL 必须是合法 URL
CAS_VALIDATE_URL 必须是合法 URL
```

### 8.2 新增路由

新增两个接口：

```txt
GET /api/auth/cas/start
GET /api/auth/cas/callback
```

#### `/api/auth/cas/start`

职责：

1. 检查 `CAS_ENABLED=true`；
2. 读取可选 `next` 参数；
3. 生成并保存 `state`；
4. 构造 CAS 登录 URL；
5. 重定向到 CAS 登录页。

示例逻辑：

```txt
CAS_LOGIN_URL?service=<CAS_SERVICE_URL>&state=<state>
```

注意：标准 CAS 通常通过 `service` 参数回跳，不一定支持 `state`。如果 CAS 不支持 state，可将 next/state 放入服务端临时 cookie，callback 时读取。

#### `/api/auth/cas/callback`

职责：

1. 读取 `ticket`；
2. 读取并校验 `state` 或服务端临时 cookie；
3. 调用 CAS validate API；
4. 解析 CAS 返回结果；
5. 获取 username/email/name/avatar；
6. 归一化用户信息；
7. 查找或创建 Multica 用户；
8. 更新用户 name/avatar；
9. 签发 JWT；
10. 设置 cookie；
11. 重定向回前端目标页面。

伪代码：

```go
func (h *Handler) CASCallback(w http.ResponseWriter, r *http.Request) {
    ticket := r.URL.Query().Get("ticket")
    if ticket == "" {
        writeError(w, http.StatusBadRequest, "missing cas ticket")
        return
    }

    casUser, err := h.validateCASTicket(r.Context(), ticket)
    if err != nil {
        writeError(w, http.StatusUnauthorized, "cas ticket validation failed")
        return
    }

    normalized, err := normalizeCASUser(casUser, h.cfg.CAS)
    if err != nil {
        writeError(w, http.StatusForbidden, "invalid cas user")
        return
    }

    user, isNew, err := h.findOrCreateUser(r.Context(), normalized.Email)
    if err != nil {
        writeError(w, http.StatusForbidden, err.Error())
        return
    }
    _ = isNew

    user = h.updateUserProfileIfNeeded(user, normalized)

    token, err := h.issueJWT(user)
    if err != nil {
        writeError(w, http.StatusInternalServerError, "failed to issue token")
        return
    }

    if err := auth.SetAuthCookies(w, token); err != nil {
        slog.Warn("failed to set auth cookies", "error", err)
    }

    redirectURL := safeNextURL(r)
    http.Redirect(w, r, redirectURL, http.StatusFound)
}
```

## 9. CAS ticket 校验

### 9.1 支持 CAS 2.0 / 3.0 XML

公司 CAS 大概率返回 XML。

典型返回：

```xml
<cas:serviceResponse xmlns:cas="http://www.yale.edu/tp/cas">
  <cas:authenticationSuccess>
    <cas:user>zhangsan</cas:user>
    <cas:attributes>
      <cas:email>zhangsan@company.com</cas:email>
      <cas:name>张三</cas:name>
    </cas:attributes>
  </cas:authenticationSuccess>
</cas:serviceResponse>
```

需要解析：

```txt
user
attributes.email
attributes.name
attributes.avatar
```

### 9.2 校验请求

请求 CAS validate：

```txt
GET CAS_VALIDATE_URL?service=<CAS_SERVICE_URL>&ticket=<ticket>
```

注意：

1. `service` 必须与 CAS 登录时的 service 完全一致；
2. scheme、host、path、query 都可能影响 CAS 校验；
3. 需要设置请求超时；
4. 不要把 ticket 打到普通日志里；
5. 校验失败返回 401。

### 9.3 用户归一化

定义内部结构：

```go
type ExternalIdentity struct {
    Provider       string
    ProviderUserID string
    Email          string
    Name           string
    AvatarURL      string
}
```

CAS 映射：

```go
ExternalIdentity{
    Provider: "cas",
    ProviderUserID: casUsername,
    Email: normalizedEmail,
    Name: casName,
    AvatarURL: casAvatar,
}
```

邮箱处理规则：

```txt
1. 优先使用 CAS attributes.email
2. 如果没有 email，则使用 username + "@" + CAS_EMAIL_DOMAIN
3. email 必须转小写并 trim
4. email 必须是合法邮箱
5. 如果配置 CAS_EMAIL_DOMAIN，则 email domain 必须匹配
```

## 10. 前端改造

### 10.1 `/api/config` 增加 auth 配置

后端 public config 增加：

```json
{
  "auth": {
    "emailLoginEnabled": false,
    "googleLoginEnabled": false,
    "cas": {
      "enabled": true,
      "displayName": "米可世界统一飞书登录",
      "loginUrl": "/api/auth/cas/start"
    }
  }
}
```

前端从 `/api/config` 读取后决定展示哪些登录方式。

### 10.2 登录页新增 CAS 按钮

在登录页增加：

```tsx
{cas?.enabled && (
  <Button
    type="button"
    variant="outline"
    className="w-full"
    size="lg"
    onClick={() => {
      const next = getSafeNextParam();
      window.location.href = `${cas.loginUrl}?next=${encodeURIComponent(next)}`;
    }}
  >
    使用{cas.displayName ?? "米可世界统一飞书登录"}登录
  </Button>
)}
```

注意：

1. 不要写死公司域名；
2. 按钮文案走 i18n；
3. 如果邮箱登录禁用，则不展示邮箱输入框；
4. 如果 Google 登录禁用，则不展示 Google 按钮；
5. 如果只有 CAS 登录，可把 CAS 按钮作为主按钮。

### 10.3 CLI callback 兼容

当前登录页有 `cliCallback` 逻辑。CAS 登录后要保证：

1. CAS callback 设置 cookie 后重定向回登录页；
2. 登录页能识别已有 cookie；
3. 登录页进入 CLI confirm 流程；
4. 用户确认后调用 `IssueCliToken`；
5. token 回传给 CLI callback。

不要让 CAS callback 直接跳转 CLI 本地地址，除非已经完整校验 callback URL 安全性。更稳妥的是：CAS 完成后回到 Multica 前端，由现有 CLI callback 逻辑处理。

## 11. 安全要求

### 11.1 next 参数校验

`next` 只能允许站内相对路径。

允许：

```txt
/workspaces/xxx
/issues
/
```

禁止：

```txt
https://evil.com
//evil.com
javascript:alert(1)
```

实现规则：

```txt
1. 必须以 / 开头
2. 不能以 // 开头
3. 不能包含协议
4. 默认回退到 /
```

### 11.2 CAS ticket 安全

要求：

1. ticket 只能使用一次；
2. validate 请求必须带超时；
3. 日志不要打印完整 ticket；
4. validate 失败必须拒绝登录；
5. CAS 返回用户为空必须拒绝登录；
6. email 为空且不能通过 username 拼接时拒绝登录。

### 11.3 Cookie 安全

生产环境：

```txt
HttpOnly=true
Secure=true
SameSite=Lax 或 Strict
```

如果浏览器直接跨子域调用 API，需要正确配置 `COOKIE_DOMAIN`。`COOKIE_DOMAIN` 为空时只对当前 host 生效，设置为 `.example.com` 时可跨子域共享 cookie，且不能设置为 IP。

推荐部署口径下，即使准备了 Web 域和 API 域，浏览器仍通过 Web 域的同源 `/api`、`/auth`、`/ws` 访问后端，API 域只给 CLI / daemon 使用。这种形态下 `COOKIE_DOMAIN` 可以留空。

### 11.4 禁用非公司登录入口

如果公司要求只能 SSO 登录：

```env
EMAIL_LOGIN_ENABLED=false
GOOGLE_LOGIN_ENABLED=false
```

并且前端不展示入口，后端接口也应拒绝调用：

```txt
POST /api/auth/send-code -> 404 或 403
POST /api/auth/verify-code -> 404 或 403
POST /api/auth/google -> 404 或 403
```

不要只在前端隐藏按钮。

## 12. 账号生命周期管理

### 12.1 当前问题

当前 `checkSignupAllowed` 对已有用户直接放行。也就是说 allowlist 只限制首次注册，不限制已有用户再次登录。

公司内部场景下，这会带来风险：

```txt
员工离职后，如果 Multica 内已有账号，单靠 ALLOWED_EMAIL_DOMAINS 无法禁用这个已有账号。
```

### 12.2 建议新增 account_status

第一阶段可以不做复杂组织同步，但建议加最小账号状态字段。

数据库增加：

```sql
ALTER TABLE "user"
ADD COLUMN account_status TEXT NOT NULL DEFAULT 'active';

ALTER TABLE "user"
ADD CONSTRAINT user_account_status_check
CHECK (account_status IN ('active', 'suspended'));
```

认证中间件和登录入口统一校验：

```txt
active     允许登录和访问
suspended  拒绝登录和访问
```

需要覆盖：

```txt
CAS callback
邮箱验证码 verify
Google OAuth callback
JWT middleware
PAT middleware
IssueCliToken
WebSocket auth
daemon 相关认证
/api/me
```

第一阶段如果时间有限，至少要在 CAS callback 和 JWT/PAT middleware 校验。

## 13. 数据库与用户字段

第一阶段不强制新增外部身份表，但建议预留。

### 13.1 最小方案

只使用现有 `user.email` 作为唯一身份。

优点：

```txt
改动小
复用现有 findOrCreateUser
与当前邮箱验证码/Google 逻辑一致
```

缺点：

```txt
如果公司账号 email 变更，用户身份会断裂
无法记录 provider_user_id
```

### 13.2 推荐增强方案

新增 `user_identities` 表：

```sql
CREATE TABLE user_identities (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
    provider TEXT NOT NULL,
    provider_user_id TEXT NOT NULL,
    email TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE(provider, provider_user_id)
);
```

CAS 登录时：

```txt
1. 先按 provider=cas + provider_user_id 查 user
2. 查不到再按 email 查 user
3. 仍查不到则创建 user
4. 写入 user_identities
```

如果希望第一阶段快速落地，可以先不做这张表。但从企业 SSO 角度，建议做。

## 15. 验收标准

### 15.1 Web 登录验收

必须通过：

```txt
1. 未登录访问 Multica，进入登录页
2. 登录页展示“公司 SSO 登录”
3. 点击后跳转公司 CAS
4. CAS 登录成功后回到 Multica
5. Multica 自动创建或查找用户
6. Web UI 登录成功
7. 刷新页面后仍保持登录
8. 退出登录后 cookie 被清除
```

### 15.2 非公司用户验收

必须通过：

```txt
1. 非公司邮箱不能创建账号
2. CAS 返回无 email 且不能拼接公司邮箱时拒绝登录
3. CAS 返回非 company.com 邮箱时拒绝登录
```

### 15.3 禁用邮箱验证码验收

当配置：

```env
EMAIL_LOGIN_ENABLED=false
```

必须满足：

```txt
1. 前端不展示邮箱验证码入口
2. POST /api/auth/send-code 不可用
3. POST /api/auth/verify-code 不可用
```

### 15.4 禁用 Google 登录验收

当配置：

```env
GOOGLE_LOGIN_ENABLED=false
```

必须满足：

```txt
1. 前端不展示 Google 登录按钮
2. Google OAuth 登录接口不可用
```

### 15.5 CLI 验收

必须通过：


## 18. 最终判断

最合适的方案是：

```txt
fork 项目
新增 CAS 登录 Provider
复用 Multica 原生 user / JWT / cookie / PAT / daemon token 链路
不要外置一套登录态后直接写 DB/缓存
```

这样改造成本可控，后续也更容易跟 upstream 合并。
