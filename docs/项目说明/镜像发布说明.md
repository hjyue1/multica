# Multica 镜像仓库与发布说明

本文整理自运维提供的飞书 Wiki，用于 Multica 迭代后构建并推送后端、前端 Docker 镜像。部署运行细节见 [VPS-部署说明.md](./VPS-部署说明.md)。

## 1. 镜像仓库信息

| 项目 | 值 |
|---|---|
| 镜像仓库实例 | `bluecity-bigdata-registry` |
| 命名空间 | `algo` |
| 当前复用仓库 | `mlops` |
| 公网/办公网推送域名 | `bluecity-bigdata-registry-registry.ap-southeast-1.cr.aliyuncs.com` |
| VPC 内网拉取域名 | `bluecity-bigdata-registry-registry-vpc.ap-southeast-1.cr.aliyuncs.com` |
| 登录用户名 | `algo-dev-acr-image@1018288888205211` |
| 密码 | 联系运维获取，不写入仓库、脚本或文档 |

运维文档里提到的内网打镜像机器是 `algo-prod-kubectl-b1`，需要使用 `guard` 用户登录，并且 Docker 需要 `sudo` 权限。

## 2. 镜像命名约定

当前运维侧暂时复用 `algo/mlops` 这个仓库。Multica 有两个运行镜像：后端和前端。为了避免两个镜像抢同一个 tag，当前采用“服务名 + 版本号”的 tag：

```text
bluecity-bigdata-registry-registry.ap-southeast-1.cr.aliyuncs.com/algo/mlops:multica-backend-<version>
bluecity-bigdata-registry-registry.ap-southeast-1.cr.aliyuncs.com/algo/mlops:multica-web-<version>
```

K8s 或 VPS 内网部署时，镜像地址应换成 VPC 域名：

```text
bluecity-bigdata-registry-registry-vpc.ap-southeast-1.cr.aliyuncs.com/algo/mlops:multica-backend-<version>
bluecity-bigdata-registry-registry-vpc.ap-southeast-1.cr.aliyuncs.com/algo/mlops:multica-web-<version>
```

后续如果运维为 Multica 新建独立仓库，建议改成：

```text
bluecity-bigdata-registry-registry-vpc.ap-southeast-1.cr.aliyuncs.com/algo/multica-backend:<version>
bluecity-bigdata-registry-registry-vpc.ap-southeast-1.cr.aliyuncs.com/algo/multica-web:<version>
```

发布脚本已支持通过 `BACKEND_REPOSITORY` 和 `WEB_REPOSITORY` 切换到独立仓库模式。

## 3. 发布脚本

脚本位置：

```bash
scripts/publish-acr-images.sh
```

默认行为：

1. 构建 `Dockerfile` 生成后端镜像；
2. 构建 `Dockerfile.web` 生成前端镜像；
3. 目标平台默认为 `linux/amd64`，适配当前 Alibaba Cloud Linux 3 / x86_64 VPS；
4. 推送到公网/办公网 ACR 域名；
5. 输出部署时应使用的 VPC 镜像地址。

本地电脑发布示例：

```bash
bash scripts/publish-acr-images.sh --login --tag v0.1.0-company.1
```

内网机器 `algo-prod-kubectl-b1` 发布示例：

```bash
DOCKER_BIN="sudo docker" \
ACR_PUSH_REGISTRY="bluecity-bigdata-registry-registry-vpc.ap-southeast-1.cr.aliyuncs.com" \
bash scripts/publish-acr-images.sh --login --tag v0.1.0-company.1
```

只预览命令，不实际构建/推送：

```bash
bash scripts/publish-acr-images.sh --dry-run --tag v0.1.0-company.1
```

生成 VPS 部署用 Compose 覆盖文件：

```bash
bash scripts/publish-acr-images.sh \
  --tag v0.1.0-company.1 \
  --write-compose-override docker-compose.images.yml

docker compose -f docker-compose.selfhost.yml -f docker-compose.images.yml up -d
```

`docker-compose.images.yml` 里会写入 VPC 域名镜像地址，适合在 VPS 或内网环境部署时使用。部署环境仍然通过 `.env` 中的远程 `DATABASE_URL` 连接 PostgreSQL，不会启动本地数据库容器。

## 4. 常用参数

| 参数/环境变量 | 说明 | 默认值 |
|---|---|---|
| `--tag <version>` / `TAG` | 发布版本号 | Git 描述或短 commit |
| `--login` | 推送前执行 `docker login` | 默认不登录 |
| `--no-push` | 只构建到本地，不推送 | 默认推送 |
| `--dry-run` | 只打印命令 | 默认关闭 |
| `--platform <platform>` / `PLATFORM` | 构建平台 | `linux/amd64` |
| `ACR_PUSH_REGISTRY` | 构建并推送使用的 registry 域名 | 公网/办公网域名 |
| `ACR_DEPLOY_REGISTRY` | 输出给部署使用的 registry 域名 | VPC 域名 |
| `ACR_NAMESPACE` | ACR 命名空间 | `algo` |
| `ACR_REPOSITORY` | 当前复用仓库名 | `mlops` |
| `BACKEND_REPOSITORY` | 后端仓库名 | 跟随 `ACR_REPOSITORY` |
| `WEB_REPOSITORY` | 前端仓库名 | 跟随 `ACR_REPOSITORY` |
| `ACR_USERNAME` | ACR 登录用户名 | `algo-dev-acr-image@1018288888205211` |
| `DOCKER_BIN` | Docker 命令，内网机可设为 `sudo docker` | `docker` |
| `REMOTE_API_URL` | Web 镜像构建时的后端代理地址 | `http://backend:8080` |
| `NEXT_PUBLIC_WS_URL` | Web 镜像构建时写入的 WebSocket 地址 | 空，浏览器按当前 origin 推导 |

切换到独立仓库模式示例：

```bash
BACKEND_REPOSITORY=multica-backend \
WEB_REPOSITORY=multica-web \
bash scripts/publish-acr-images.sh --login --tag v0.1.0-company.1
```

这时镜像会变成：

```text
.../algo/multica-backend:v0.1.0-company.1
.../algo/multica-web:v0.1.0-company.1
```

## 5. 建议发布流程

1. 合并或切到要发布的代码分支；
2. 本地执行 `pnpm typecheck`、`pnpm test`、`make test`，必要时执行 `make check`；
3. 确认版本号，例如 `v0.1.0-company.1`；
4. 执行 `bash scripts/publish-acr-images.sh --login --tag <version>`；
5. 在部署环境使用脚本输出的 VPC 镜像地址更新 Compose/K8s 配置；
6. 部署后检查：

```bash
curl -f http://127.0.0.1:8080/health
curl -f http://127.0.0.1:8080/readyz
```

## 6. 回滚

回滚时不要重新打同名 tag。直接把 Compose/K8s 配置里的镜像地址改回上一个已验证版本，例如：

```text
bluecity-bigdata-registry-registry-vpc.ap-southeast-1.cr.aliyuncs.com/algo/mlops:multica-backend-v0.1.0-company.0
bluecity-bigdata-registry-registry-vpc.ap-southeast-1.cr.aliyuncs.com/algo/mlops:multica-web-v0.1.0-company.0
```

然后重新执行部署命令。
