# 免邮件邀请自动加入

> 创建时间：2026-05-17

## 功能概述

公司 SSO 部署场景下，工作区管理员可以只录入员工邮箱，不发送邀请邮件。员工后续通过公司 SSO 登录时，系统会按邮箱自动接受 pending invitation 并加入对应工作区。

## 技术方案

### 核心实现

新增两个环境变量控制邀请行为：

```env
INVITATION_EMAIL_ENABLED=false
INVITATION_AUTO_ACCEPT_ON_LOGIN=true
```

`INVITATION_EMAIL_ENABLED=false` 时，`CreateInvitation` 只创建 `workspace_invitation` 记录，不调用 `EmailService.SendInvitationEmail`。`INVITATION_AUTO_ACCEPT_ON_LOGIN=true` 时，登录链路在用户创建或查找完成后调用自动接受逻辑：查询该用户邮箱匹配的 pending invitation，事务内标记 accepted、创建 `member`、标记用户已完成 onboarding，并广播成员加入事件。

该逻辑复用现有 invitation 表和 accept 语义，不新增数据库表。

## API / 配置

### 环境变量

| 变量 | 默认 | 说明 |
|------|------|------|
| `INVITATION_EMAIL_ENABLED` | `true` | 是否发送邀请邮件 |
| `INVITATION_AUTO_ACCEPT_ON_LOGIN` | `false` | 登录后是否自动接受匹配邮箱的 pending invitation |

### `/api/config`

新增字段：

```json
{
  "auth": {
    "invitation_email_enabled": false,
    "auto_accept_invitations_on_login": true
  }
}
```

## 设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 是否直接创建 member | 否 | 用户首次登录前可能没有 `user_id`，无法写入成员关系 |
| 是否新增表 | 否 | 现有 `workspace_invitation` 已能表达“邮箱预授权” |
| 是否保留邮件能力 | 是 | 通过环境变量控制，默认不破坏原邮件邀请行为 |
| 自动加入时机 | 登录成功后 | 此时已有用户记录和邮箱，能安全创建成员关系 |

## 已知限制与待改进

- 管理员输入的邮箱必须和 CAS 登录后归一化得到的邮箱一致。
- 如果 CAS `<cas:user>` 返回的是工号而不是邮箱，需要改用 CAS email attribute 或配置 `CAS_EMAIL_DOMAIN`。
- 当前自动接受只在登录链路触发；已登录用户被补录邀请后，需要重新登录或后续增加主动刷新/一键加入入口。
